F FON Banking / Architecture
Decision Ownership Implementation Migration

Technical proposal · July 2026

One app.
Two kinds of state.

A practical state-management and data-fetching strategy for the FON Banking Expo app—designed around ownership, security, predictable caching, and a codebase that stays easy to change.

Short answer Your choice is right: TanStack Query + Zustand, with strict and intentionally small boundaries.
01 / DECISION

Keep the pairing. Refine the responsibility.

The libraries are complementary, not competing. The architecture succeeds when the same value never has two owners.

Adopt TanStack Query v5 for server state. Add Zustand v5 only for genuinely shared client state.

Accounts, cards, transactions, payment templates, profile data, request status, errors, retries, freshness, and invalidation belong to TanStack Query. A draft payment that spans screens, a privacy-mode toggle, onboarding progress, or a global sheet state may belong to Zustand. Input used by only one screen stays in React state.

02 / OWNERSHIP

Put every value in exactly one place.

This boundary is more important than the library choice. It prevents stale copies, manual synchronization, and broad re-renders.

TanStack Query

Server-owned truth

Anything fetched from or written to the banking API, plus the lifecycle around those requests.

  • Accounts, cards, transactions
  • Loading, error, stale & retry state
  • Pagination and background refresh
  • Mutation status and invalidation
  • Cached derived views via select
Zustand

Shared client state

State created by the app that several distant screens need and the server does not own.

  • Multi-screen payment draft
  • Selected source account
  • Balance privacy preference
  • Transient cross-route UI state
  • Non-secret user preferences
React / Router

Local & navigation state

Keep state close to where it is rendered. Global does not mean convenient; it means shared ownership.

  • Input text and validation display
  • Open/closed local modal
  • Animation and swiper position
  • Route IDs and search parameters
  • One-screen filter selection
The placement test: fetched from the API → Query; shared across routes but not server-owned → Zustand; otherwise → local state or route params.
03 / TODAY

What the current hook gets right—and what it cannot scale.

useBankingData is a useful first boundary, but every mounted consumer owns a separate request cycle and private copy of the same data.

Concern Current approach Recommended approach Why it matters
Shared results Per-hook local state Query cache keyed by resource Several screens reuse one result instead of issuing duplicate requests.
Freshness Manual mount effect staleTime + app-focus refresh Data is refreshed intentionally when the app returns to the foreground.
Errors & retries One combined error Per-resource policies An accounts failure is different from a card or transaction failure.
Writes No cache strategy Mutations + targeted invalidation A transfer can refresh the affected account and transaction keys only.
Logout Local reset per mount queryClient.clear() Private data leaves memory once, centrally, when the session ends.
Auth header Two Axios creation paths One client + request interceptor Removes drift between the context client and useApi.

03 / TRANSPORT

Typed API modules

A single Axios client converts HTTP responses and errors into domain-friendly values.

Zustand sits beside this flow.It supplies client-only values such as a draft or preference; it does not mirror the query cache.
SecureStore sits below authentication.Keep secrets and small identifiers there. Do not persist banking payloads in a general client store.

export const accountKeys = {
  all: ["accounts"] as const,
  detail: (id: string) =>
    ["accounts", id] as const,
  cards: (id: string) =>
    ["accounts", id, "cards"] as const,
  transactions: (id: string) =>
    ["accounts", id, "transactions"] as const,
};

export function useAccounts() {
  return useQuery({
    queryKey: accountKeys.all,
    queryFn: accountsApi.list,
  });
}
features/accounts/queries.ts● ● ● 
export function useAccountCards(accountId?: string) {
  return useQuery({
    queryKey: accountKeys.cards(accountId ?? ""),
    queryFn: () => accountsApi.cards(accountId!),
    enabled: Boolean(accountId),
  });
}

export function useAccountTransactions(id?: string) {
  return useInfiniteQuery({
    queryKey: accountKeys.transactions(id ?? ""),
    queryFn: ({ pageParam }) =>
      accountsApi.transactions(id!, pageParam),
    initialPageParam: 1,
    enabled: Boolean(id),
    getNextPageParam: (page) => page.nextPage,
  });
}

type PaymentDraft = {

⚿

Banking-specific security boundary

Keep session secrets in Expo SecureStore, clear the query cache on logout or user change, and avoid persisting balances, card data, CVV values, or transaction history unless an explicit offline-security design has been approved. A persisted cache is a product and threat-model decision—not a default optimization.

06 / POLICY

Sensible defaults, deliberate exceptions.

Banking data should feel current without turning every tab change into a network storm.

Resource Suggested freshness Refresh trigger Notes
Accounts / balances 15–30 seconds App focus, pull-to-refresh, successful transfer Invalidate affected accounts after money movement.
Recent transactions 15–30 seconds App focus, pull-to-refresh, successful transfer Prefer a paginated backend endpoint over loading every account’s full history.
Cards 5 minutes App focus when stale, card mutation Card metadata changes less frequently than balances.
Payment templates 5 minutes Template create/update/delete Optimistic updates are reasonable if rollback is clear.
Reference data 30 minutes+ Reconnect or manual invalidation Currency lists and payment codes can remain fresh longer.
Retry rule: retry transient reads once or twice; do not automatically retry money-moving mutations unless the backend provides an idempotency key and the flow is explicitly designed for it.

“Queries” and “stores” are implementation details. Accounts, payments, cards, and auth are the concepts developers navigate.

One direction of dependency

Screens import feature hooks. Feature hooks import typed API functions and query keys. The API layer imports the single Axios client. Stores do not import query data or copy it into their own fields.

If the current top-level folders are preferred, the same boundary still works. The architectural rule matters more than adding a src directory.

09 / GUARDRAILS

What not to build.

Most state-management complexity comes from duplicated ownership, not insufficient tooling.

Avoid Because Use instead
Copying query results into Zustand Creates two truths and synchronization bugs. Read directly from Query; derive with select.
One giant “app store” Couples unrelated domains and encourages broad subscriptions. Small purpose-built stores with narrow selectors.
Persisting the whole query cache Leaves sensitive, stale banking data on device. Start memory-only; approve offline scope explicitly.
Global state for every form field Adds lifecycle and reset complexity without sharing value. Local form state; Zustand only for cross-route drafts.
Blind mutation retries Can duplicate financial actions. No retries by default; use server idempotency keys.
A single aggregate “load everything” query Couples failure, freshness, and rendering of unrelated resources. Composable resource queries with independent keys.

Architecture decision

Approve Zustand + TanStack Query—with TanStack Query doing most of the work.

Your proposed stack is the best fit for this Expo app. The correction is conceptual: do not think of Zustand as the home of “global data.” Treat TanStack Query as the owner of remote banking data and Zustand as a small tool for shared, client-created state. React state and Expo Router complete the picture for everything local or navigational.

Outcome: less custom loading code, fewer duplicate requests, explicit security boundaries, and simpler screens.
