Decision Ownership Implementation Migration

Technical proposal В· July 2026

One value.
One owner.

A practical state-management and data-fetching strategy for the FON Banking Expo appвЂ”designed around ownership, security, predictable caching, and a codebase that stays easy to change.

Short answer Use TanStack Query for all banking API data. There is no current use case for Zustand in this app.
01 / DECISION

Do not add Zustand.

The app has no shared, client-created state that needs a global store. Adding one now would create another possible owner without solving a current problem.

Adopt TanStack Query v5 as the only in-memory owner of banking API data. Keep UI state local, navigation state in Expo Router, and the session boundary in AuthContext plus SecureStore.

Accounts, cards, transactions, payment templates, profile data, request status, errors, retries, freshness, and invalidation belong to TanStack Query. The current payment form is one screen, so its fields and selected account remain local React state. Product IDs and scanned QR results cross routes as serializable route params. None of these values needs a second global store.

02 / OWNERSHIP

Put every value in exactly one place.

This boundary is more important than the library choice. It prevents stale copies, manual synchronization, and broad re-renders.

TanStack Query

Server-owned truth

Anything fetched from or written to the banking API, plus the lifecycle around those requests.

  • Accounts, cards, transactions
  • Loading, error, stale & retry state
  • Pagination and background refresh
  • Mutation status and invalidation
  • Cached derived views via select
AuthContext + SecureStore

Session state

Authentication lifecycle is a small app-wide concern. Secrets and device identifiers stay behind the existing auth boundary.

  • Authentication status
  • In-memory session token
  • Login, logout, activation and PIN actions
  • Device ID and PIN-setup marker
  • Query-cache cleanup on logout
React / Router

Local & navigation state

Keep state close to where it is rendered. Global does not mean convenient; it means shared ownership.

  • Input text and validation display
  • Open/closed local modal
  • Animation and swiper position
  • Route IDs and search parameters
  • One-screen filter selection
The placement test: fetched from or written to the API в†’ TanStack Query; session secret or auth lifecycle в†’ AuthContext and SecureStore; encoded in navigation в†’ route params; otherwise в†’ local component state. Never copy a query result into another store.
03 / TODAY

The app audit finds no global client-state use case.

useBankingData is a useful first boundary, but every mounted consumer owns a separate request cycle and private copy of the same data. That duplication calls for a query cache, not a second client store.

Concern Current approach Recommended approach Why it matters
Shared results Per-hook local state Query cache keyed by resource Several screens reuse one result instead of issuing duplicate requests.
Freshness Manual mount effect staleTime + app-focus refresh Data is refreshed intentionally when the app returns to the foreground.
Errors & retries One combined error Per-resource policies An accounts failure is different from a card or transaction failure.
Writes No cache strategy Mutations + targeted invalidation A transfer can refresh the affected account and transaction keys only.
Logout Local reset per mount queryClient.clear() Private data leaves memory once, centrally, when the session ends.
Auth header Two Axios creation paths One client + request interceptor Removes drift between the context client and useApi.
Whole-app audit: auth-screen inputs, the payment form and account menu, transaction search/filter/page, scanner progress, switches, and carousel positions are all local UI state. Static exchange-rate constants remain ordinary module data. Product IDs already belong to the router. Authentication is already shared through AuthContext. The only unusual cross-route value is the QR scan result, currently returned through a module-level callback; make it a serializable route param instead of introducing a store. Result: zero current use cases for a global client-state library.

features/accounts/queries.tsв—Џ в—Џ в—Џ 
import { queryOptions, useQuery } from "@tanstack/react-query";

const accountsQuery = () => queryOptions({
  queryKey: ["accounts"] as const,
  queryFn: accountsApi.list,
  staleTime: 30_000,
});

export function useAccounts() {
  return useQuery(accountsQuery());
}

export function useAccount(accountId?: string) {
  return useQuery({
    ...accountsQuery(),
    select: (accounts) =>
      accounts.find((account) => account.accountId === accountId),
    enabled: Boolean(accountId),
  });
}

// Both hooks subscribe to the exact same ["accounts"] cache entry.
// select changes the view returned to a screen, not the cached truth.

Banking-specific security boundary

Keep session secrets in Expo SecureStore, clear the query cache on logout or user change, and avoid persisting balances, card data, CVV values, or transaction history unless an explicit offline-security design has been approved. A persisted cache is a product and threat-model decisionвЂ”not a default optimization.

06 / POLICY

Sensible defaults, deliberate exceptions.

Banking data should feel current without turning every tab change into a network storm.

Resource Suggested freshness Refresh trigger Notes
Accounts / balances 15вЂ“30 seconds App focus, pull-to-refresh, successful transfer Invalidate affected accounts after money movement.
Recent transactions 15вЂ“30 seconds App focus, pull-to-refresh, successful transfer Prefer a paginated backend endpoint over loading every accountвЂ™s full history.
Cards 5 minutes App focus when stale, card mutation Card metadata changes less frequently than balances.
Payment templates 5 minutes Template create/update/delete Optimistic updates are reasonable if rollback is clear.
Reference data 30 minutes+ Reconnect or manual invalidation Currency lists and payment codes can remain fresh longer.
Retry rule: retry transient reads once or twice; do not automatically retry money-moving mutations unless the backend provides an idempotency key and the flow is explicitly designed for it.

One direction of dependency

Screens import feature hooks. Feature hooks import typed API functions and canonical query options. The API layer imports the single Axios client. No client data-store folder is needed, and no layer copies query data into React context or local state.

If the current top-level folders are preferred, the same boundary still works. The architectural rule matters more than adding a src directory.

08 / MIGRATION

Move in small, testable slices.

No rewrite is needed. Introduce the cache at the root, then replace the current aggregate hook one resource at a time.

Install and configure TanStack Query

Add one stable QueryClient at the root; wire React Native AppState and network status.

foundation

Unify the Axios client

Move base URL, auth header injection, and normalized error mapping into one transport module.

small

Migrate accounts first

Create account keys, API functions, and useAccounts; switch the home and products screens.

Split cards and transactions by account

Use dependent or parallel queries. Add backend pagination before transaction history grows.

medium

high care

09 / GUARDRAILS

What not to build.

Most state-management complexity comes from duplicated ownership, not insufficient tooling.

Avoid Because Use instead
Copying query results into context or local state Creates two truths and synchronization bugs. Read directly from Query; derive with select.
Adding a global client store without a current owner problem Adds dependencies, reset rules, and another place to put API data by mistake. React state, route params, and the existing auth context.
Persisting the whole query cache Leaves sensitive, stale banking data on device. Start memory-only; approve offline scope explicitly.
Global state for every form field Adds lifecycle and reset complexity without sharing value. Local form state; serializable route params when routes communicate.
Blind mutation retries Can duplicate financial actions. No retries by default; use server idempotency keys.
A single aggregate вЂњload everythingвЂќ query Couples failure, freshness, and rendering of unrelated resources. Composable resource queries with independent keys.

Architecture decision

Approve TanStack Query. Do not add Zustand.

The full frontend contains no shared client-created state that justifies a global store. Treat TanStack Query as the sole in-memory owner of remote banking data, React as the owner of screen-local UI, Expo Router as the owner of route identity and serializable cross-route values, and AuthContext plus SecureStore as the session boundary. вЂњSingle source of truthвЂќ means one owner per valueвЂ”not one giant query that loads every resource together.

Outcome: less custom loading code, fewer duplicate requests, explicit security boundaries, and simpler screens.
